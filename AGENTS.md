# AGENTS.md

## Первичная настройка

- Если пользователь дал ссылку на этот публичный шаблон и попросил создать свой личный репозиторий, используй `skills/create-private-workspace/`: создай новый приватный repository from template, не fork, затем продолжи настройку уже в созданной копии.
- До содержательной работы проверь наличие `PROFILE.md`, `workspace.json` и `.local/machine-setup.json`.
- Если `PROFILE.md` или `workspace.json` отсутствует либо `workspace.json` содержит `"initialized": false`, обязательно используй навык `skills/setup-workspace/` и следуй `SETUP.md`.
- Если отсутствует только `.local/machine-setup.json`, выполни машинную часть навыка `setup-workspace`: проверь локальные пути и доступы, не меняя профиль владельца без необходимости.
- Не записывай персональные данные, пока не подтверждено, что рабочая копия является приватной или полностью локальной. При публичной либо неизвестной видимости останови персонализацию и объясни риск владельцу.

## Автоматические обновления

- Если существует настроенный `workspace.json` и `updates.autoCheck=true`, перед первой содержательной работой за день выполни `python3 scripts/template_update.py --if-due`.
- Код `10` означает доступное обновление: сразу используй `skills/update-workspace-template/` и выполни `python3 scripts/template_update.py --auto`, не запрашивая отдельное подтверждение при `updates.autoApply=true`.
- Обновляйся из exact `workspace.json.template.source`; не подменяй источник ближайшим fork или другим репозиторием.
- Выполняй обновление только в отдельном worktree. Не изменяй текущую ветку до успешных миграций, тестов и privacy scan.
- При `autoCommit=true` самостоятельно создай update-коммит; при `autoPush=true` отправь его только в подтверждённый private origin.
- Остановись только при небезопасном конфликте, проваленных проверках, неподтверждённом источнике/checksum или `requiresUserAction=true` в manifest. Грязное рабочее дерево означает отложить обновление до следующего чистого старта, а не просить пользователя вручную разбирать его.

При подготовке нового release следуй `RELEASES.md`. Сохраняй все ещё необходимые миграции в `migrationCommands`: пользователь может перейти сразу через несколько версий, а каждая миграция должна быть повторно запускаемой. После всех изменений согласуй `VERSION`, `workspace.example.json`, текущую версию в `README.md` и manifest version, последней операцией над содержимым выполни `python3 scripts/build_template_manifest.py --write`, затем полный validation. Tag `vX.Y.Z` обязан указывать на релизный коммит `main`, название GitHub Release – `X.Y.Z`, а описание – содержать обязательные секции `Что нового` и `Обновление`.

## Рабочий контекст

- Перед работой читай этот файл и `PROFILE.md`, если он существует.
- `PROFILE.md` хранит только устойчивые факты и предпочтения владельца. Процедуры и правила выполнения держи в `AGENTS.md`, `routines/` или навыках.
- Не превращай косвенные выводы в подтверждённые факты. Сохраняй их как гипотезы или вопросы.
- Если пользователь прислал файл без пояснений, самостоятельно исследуй его, найди подходящую задачу или проект и предложи обоснованную привязку вместо формального вопроса «что с ним делать».

## Агентские чаты

- Если текущая агентская среда умеет переименовывать разговоры и текущий разговор можно определить надёжно, после первичного сбора контекста дай ему короткое информативное название. Не оставляй сырой заголовок первого сообщения вроде `Проверить задачу`, если тему можно назвать по объекту и действию, например `Документы: проверка договора` или `Поездка: подбор билетов`.
- Считай переименование одноразовой нормализацией, а не обязанностью обновлять заголовок после каждого сообщения. Не переименовывай разговор повторно, если его заголовок уже достаточно информативен; исключение – прямое поручение владельца или существенный поворот темы, из-за которого прежнее название стало вводить в заблуждение.
- Не добавляй в заголовок номера внутренних задач, идентификаторы разговоров или тредов, URL, хеши коммитов, длинные номера документов и другие служебные значения, если без них можно различить разговоры. Если идентификатор является естественной частью названия объекта или необходим для различения похожих разговоров, используй самый короткий читаемый вариант.
- Переименовывай только текущий разговор. Предпочитай инструмент, который явно действует на текущий разговор без ручного выбора идентификатора. Если приходится искать разговор через список, проверяй одновременно несколько признаков: соответствие свежего предпросмотра последнему запросу, совпадение рабочей папки, активность или недавнее обновление и отсутствие другого правдоподобного кандидата. Не полагайся только на один признак.
- Если среда не поддерживает названия разговоров, подходящий инструмент недоступен или текущий разговор нельзя безопасно отличить от других, пропусти переименование. Не устанавливай обходную интеграцию и не угадывай идентификатор только ради заголовка; кратко сообщай об ограничении лишь тогда, когда оно существенно для запроса пользователя.

## Приватность и безопасность

- Пароли, токены, cookies, приватные ключи, TOTP-секреты и браузерные профили по умолчанию храни только в `.local/` или системном хранилище секретов.
- Не добавляй секреты в Git и не публикуй их во внешние сервисы без отдельного осознанного решения владельца.
- Если владелец включил `encrypted-recovery`, добавляй новый постоянный ignored-секрет в точный scope `recovery-plan.json` только после проверки пути и согласия владельца. До завершения работы создай и проверь новый зашифрованный backup этого scope; не добавляй весь `.local/` одной записью.
- Перед коммитом проверяй состав индекса, крупные файлы, абсолютные локальные пути и возможные секреты.
- Не маскируй и не изменяй документы автоматически, если это нарушит их доказательную или практическую ценность. Для внешней передачи готовь отдельную обезличенную копию.
- Не выполняй внешние отправки, оплаты, публикации, юридически значимые действия или изменения учётных записей без ясного разрешения пользователя.

## Структура

- `tasks/ГОД/NNNN--slug/` – отдельная задача, её контекст, журнал, источники и результат.
- `projects/slug/` – долгосрочное направление, объединяющее несколько задач.
- `routines/slug/` – повторяемая процедура.
- `entities/` – рабочий справочник людей и организаций.
- `templates/` – шаблоны новых объектов.
- `skills/` – переносимые агентские навыки без персональных данных и доступов. Codex читает их по правилам этого файла, а Claude Code получает те же правила через `CLAUDE.md`.
- `shared/` – ignored-каталог отдельных Git checkout общих объектов; декларации хранятся в `workspace.json`, а в исходных логических путях остаются tracked references.
- `.local/` – ignored-данные конкретного компьютера.
- `recovery-plan.json` – опциональный tracked allowlist точных `.local/`-путей для зашифрованного восстановления; значений секретов и recovery-пароля в нём нет.

Если владелец выбрал локальную папку для крупных или внешних файлов, используй `external-file-storage`. Провайдер синхронизации не важен: Google Drive, Яндекс Диск, Dropbox, OneDrive, Syncthing, NAS и обычная локальная папка обрабатываются одинаково. Абсолютный путь храни только в ignored `.local/external-file-storage.json`, а в `external-assets.yml` рядом с задачей или проектом записывай относительные пути и метаданные восстановления. Для backup ignored-секретов используй поверх этой папки `encrypted-recovery`: внешнее хранилище должно получать только проверенные зашифрованные архивы.

## Организация работы

- Для содержательного вопроса, к которому полезно вернуться, используй отдельную задачу. Не создавай задачу для одноразового ответа без будущей ценности.
- Перед созданием новой папки ищи существующую задачу или проект по теме.
- Для задач используй навык `manage-personal-task`, для долгосрочных направлений – `manage-personal-project`, для необъяснённых входящих файлов – `process-incoming-file`.
- Когда пользователь просит поделиться проектом, подпроектом, задачей или папкой, используй `share-workspace-object`. Когда просит принять общий объект, используй `accept-workspace-share`.
- Общий объект всегда является отдельным private repository с `workspace-module.json`; папка, ветка и sparse checkout родительского репозитория не считаются границей доступа.
- Стандартный режим истории общего объекта – fresh snapshot. Не экспортируй историю личного родительского Workspace без отдельной явной просьбы и расширенной проверки приватности.
- Фиксируй источник, дату получения, проверенные факты, открытые вопросы, принятое решение и следующий шаг.
- Не смешивай тяжёлые бинарные файлы с переносимым текстовым контекстом без необходимости.

## Сервисные навыки

- Перед первым использованием внешнего сервисного навыка выполни `python3 scripts/integration_preflight.py ИМЯ`. Для локального `encrypted-recovery` вместо этого используй его собственные команды `init` и `status`.
- Если интеграция ещё не включена, явный запрос пользователя на работу с этим сервисом разрешает добавить её в `workspace.json`, но не разрешает сохранять секреты или выполнять финальное внешнее действие.
- Используй существующую авторизованную сессию Chrome, когда она нужна для личного кабинета. Иначе используй доступный безопасный браузер и передай владельцу первый вход.
- Всегда останавливайся на CAPTCHA, SMS/push, TOTP, электронной подписи, восстановлении доступа и неожиданной проверке личности.
- Для `email-mailbox`, `external-file-storage`, `encrypted-recovery`, `gosuslugi`, `gas-pravosudie`, `russian-post-registered-mail`, `max-messenger`, `telegram-messenger`, `ozon-buyer-search`, `trelio` и `t-bank` следуй соответствующему `skills/<name>/SKILL.md`.
- Почтовые аккаунты, OAuth-токены и пароли внешних приложений храни только в `.local/email/accounts.toml` с правами `600`. Для чтения предпочитай API провайдера и используй IMAP как настроенный резервный транспорт; для отправки предпочитай API и используй SMTP как настроенный резервный транспорт.
- Т-Банк в публичном шаблоне предназначен только для явно запрошенного чтения. Платежи, переводы и изменения банковского профиля запрещены.

## Git

- Коммить только законченные логические изменения и только после проверки diff.
- Не используй `git add .` вслепую.
- Не публикуй и не меняй remote без явной команды владельца.
- Если рабочее дерево уже содержит чужие изменения, не включай их в свой коммит.
- После правок запускай подходящие проверки; минимум – `git diff --check` и `python3 scripts/check_workspace.py --mode configured` после завершённой первичной настройки.

## Исправления центрального шаблона

- При обнаружении возможного бага сначала отдели универсальный дефект шаблона от локальной настройки, данных, аккаунта или окружения владельца.
- Для универсального бага используй `skills/contribute-template-fix/` и `CONTRIBUTING.md`.
- Значение `contributions.allowDraftPullRequests=true` разрешает без отдельного подтверждения создать обезличенную публичную ветку/fork и draft PR после чистого воспроизведения, тестов и privacy scan, если агент обнаружил исправление сам и владелец не поручал публикацию PR.
- Если владелец прямо поручил подготовить или опубликовать PR в центральный шаблон, после успешных проверок открывай ready PR, если владелец не попросил оставить его черновиком. Draft уже является публичным и нужен для незавершённой работы либо как граница автономной инициативы агента, а не как дополнительная защита от утечки.
- Никогда не используй ветку, историю или pull request из личного private repository. Работай в отдельной чистой копии центрального шаблона.
- Не создавай публичный issue по умолчанию, не делай merge и не выпускай релиз без отдельного разрешения.
- Возможную уязвимость не публикуй: используй приватный GitHub Security Advisory или остановись и передай владельцу безопасный отчёт.
