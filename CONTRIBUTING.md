# Исправления центрального шаблона

Центральный репозиторий: `https://github.com/ivaschru/personal-codex-workspace`.

## Что следует возвращать

Draft PR подходит, если проблема:

- воспроизводится на чистой версии центрального шаблона;
- не зависит от личности, аккаунта, локального пути, частного сервиса или предпочтения владельца;
- может затронуть других пользователей шаблона;
- имеет минимальное исправление и проверку от повторного появления.

Локальную конфигурацию, персональные правила, данные задач и частные интеграции исправляйте только в личном репозитории.

## Безопасный порядок

1. Зафиксировать версию шаблона и минимальное воспроизведение.
2. В отдельной ignored-папке клонировать или fork-нуть центральный публичный репозиторий.
3. Воспроизвести дефект без копирования личной задачи, профиля или истории private repository.
4. Создать ветку `fix/<short-slug>` и минимальное исправление.
5. Добавить тест либо машинную проверку, которая падала до исправления.
6. Выполнить template validation, тесты, `git diff --check` и privacy scan.
7. Проверить staged diff и историю ветки вручную средствами агента.
8. Открыть PR с симптомом, воспроизведением, причиной, исправлением и проверками. Статус выбрать по правилам ниже.

Если у пользователя нет права записи в центральный репозиторий, используется отдельный публичный fork только центрального шаблона. Личный private repository и его история никогда не становятся fork или head-веткой PR.

## Публичные действия

`contributions.allowDraftPullRequests=true` разрешает агенту по собственной инициативе создать публичную ветку/fork и draft PR после всех проверок. Draft PR уже публичен, но показывает, что владелец ещё не поручал представить результат как готовый к ревью и слиянию.

Если владелец прямо поручил подготовить или опубликовать PR в центральный шаблон, полностью проверенный PR открывается сразу в статусе ready, если владелец не попросил оставить его черновиком. Простая просьба исправить локальную проблему не считается разрешением на публичный ready PR: из контекста должно быть ясно, что поручение относится именно к центральному шаблону и публикации PR.

Ни автономный draft PR, ни явно порученный ready PR не разрешают:

- создавать публичные issues;
- делать merge;
- выпускать release;
- публиковать данные из личного репозитория.

## Уязвимости

Подозрение на утечку секретов, обход защитных границ или иной security defect нельзя описывать в публичном PR или issue до исправления. Используйте приватный GitHub Security Advisory центрального репозитория. Если приватный канал недоступен, остановитесь и передайте владельцу локальный отчёт без публикации деталей.
