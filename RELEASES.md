# Выпуск релизов

Этот документ задаёт единый пользовательский формат GitHub Releases и обязательную проверку перед публикацией.

## Название и тег

- Название релиза – точная версия без префикса: `1.5.0`.
- Git tag – та же версия с префиксом `v`: `v1.5.0`.
- `VERSION`, `workspace.example.json`, версия в `template-manifest.json`, строка текущей версии в `README.md` и тег обязаны совпадать.

## Описание

Каждый релиз использует две обязательные секции. Секция `Важно` добавляется между ними только при несовместимости, ручном действии или существенном риске.

```markdown
## Что нового

- Основное изменение для владельца Workspace.
- Дополнительная возможность или важное исправление.

## Важно

- Обязательное действие или ограничение. Удалить секцию целиком, если её содержание пусто.

## Обновление

- Установка: автоматически через stable-обновление.
- Действия владельца: не требуются.
- Миграции: отсутствуют.
```

В `Что нового` оставлять от трёх до шести коротких пользовательских пунктов. Не переносить сюда список коммитов, номера PR, хеши, внутренние пути и полный технический журнал: они остаются в PR, Actions и `CHANGELOG.md`.

В `Обновление` всегда явно указывать способ установки, действия владельца и миграции. Если обновление ручное, назвать точную команду или документ. Если миграция автоматическая, кратко описать, что она меняет.

Секцию `Проверки` в GitHub Release не добавлять: результаты CI и технические проверки относятся к релизному PR. Исключение – проверка, результат которой непосредственно важен владельцу при принятии решения об установке.

## Порядок выпуска

1. Обновить `CHANGELOG.md`, `VERSION`, `workspace.example.json`, `README.md` и версию manifest.
2. Сохранить все ещё необходимые идемпотентные миграции в `migrationCommands`.
3. Последней операцией над содержимым выполнить `python3 scripts/build_template_manifest.py --write`.
4. Выполнить полный template validation, тесты, `git diff --check`, privacy scan и проверку крупных файлов.
5. Слить проверенный релизный PR.
6. Создать аннотированный тег `vX.Y.Z` на точном коммите `main` и отправить его.
7. Создать GitHub Release с названием `X.Y.Z` и описанием по шаблону выше.
8. Проверить, что API `releases/latest` возвращает новый тег и что тег указывает на текущий релизный коммит `main`.
