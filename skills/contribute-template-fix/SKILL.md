# Исправление центрального шаблона

## Классифицировать проблему

Прочитать `CONTRIBUTING.md`, `workspace.json` и текущий `VERSION`.

- **Локальная проблема:** зависит от профиля, данных, аккаунта, пути, ОС-настройки или предпочтения владельца. Исправить только локально.
- **Универсальный баг:** воспроизводится в чистом центральном шаблоне и может затронуть других. Продолжить этот workflow.
- **Предложение:** не называть багом; подготовить локальное предложение, но не создавать публичный issue при `allowPublicIssues=false`.
- **Уязвимость:** не создавать публичный PR/issue; использовать приватный GitHub Security Advisory или остановиться с локальным отчётом.

## Воспроизвести отдельно

1. Получить URL из `workspace.json.template.source`, fallback – URL из `CONTRIBUTING.md`.
2. Создать отдельную чистую копию центрального репозитория в ignored `.local/template-contribution/<slug>/` или вне личного репозитория.
3. Не использовать личный private repository как upstream, fork, PR head или источник истории.
4. Зафиксировать минимальные шаги и убедиться, что дефект существует на актуальной центральной ветке.

Если чистое воспроизведение не получилось, оставить исправление локальным и не публиковать предположение как баг шаблона.

## Исправить

1. Создать ветку `fix/<short-slug>` от центральной default branch.
2. Реализовать минимальное общее исправление без личных значений.
3. Добавить регрессионный тест или детерминированную проверку.
4. Запустить проверки репозитория, `git diff --check` и privacy scan.
5. Проверить staged diff, список файлов, крупные/binary-файлы и всю историю новой ветки.

Запрещено переносить `PROFILE.md`, `workspace.json` владельца, `.local/`, задачи, проекты, документы, абсолютные пути, cookies, токены, session-файлы и фрагменты личных источников.

## Опубликовать PR

Прочитать `workspace.json.contributions`.

- Если владелец не поручал публикацию PR, при `allowDraftPullRequests=false` подготовить patch и описание локально, затем запросить разрешение.
- Если владелец не поручал публикацию PR, при `allowDraftPullRequests=true` отдельное подтверждение не требуется: можно создать публичный fork центрального шаблона при отсутствии write-доступа, push ветки и draft PR.
- Если владелец прямо поручил подготовить или опубликовать PR именно в центральный шаблон, после всех проверок открыть ready PR, если владелец не попросил оставить его черновиком. Общая просьба исправить локальную задачу сама по себе не разрешает публичную публикацию.
- Не создавать fork личного репозитория. Fork должен происходить только от центрального публичного шаблона.
- В PR описать симптом, чистое воспроизведение, причину, исправление и выполненные тесты; не упоминать личные сущности.
- После создания перечитать PR и дать владельцу прямую ссылку.

Не делать merge, не изменять release и не создавать public issue без отдельного разрешения. Draft PR уже публичен: его статус не заменяет privacy scan и не защищает от утечки.
