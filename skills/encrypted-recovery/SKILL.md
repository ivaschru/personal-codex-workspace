# Зашифрованное восстановление

## Безопасная модель

- Использовать только парольный режим `age`: он обеспечивает аутентифицированное шифрование и не требует собственной криптографической реализации Workspace.
- Не сохранять recovery-пароль в Git, `workspace.json`, плане, архиве или внешнем хранилище. Предложить владельцу сохранить его в менеджере паролей.
- Добавлять в план только точные относительные пути внутри `.local/`. Не архивировать весь ignored-каталог или `.gitignore` целиком.
- Считать backup успешным только после автоматической обратной расшифровки и проверки SHA-256.
- Не удалять исходные секреты после backup.
- Для disaster recovery выбирать папку на другом устройстве или с подтверждаемой внешней синхронизацией. Другая папка того же диска защищает от случайного удаления, но не от потери устройства.

## Настроить

1. Подтвердить, что Workspace имеет видимость `private` или `local-only`.
2. Настроить `external-file-storage`, если локальная папка ещё не выбрана.
3. Проверить наличие `age`; если его нет, остановиться и предложить штатную установку для ОС владельца.
4. Выполнить:

```bash
python3 skills/encrypted-recovery/scripts/encrypted_recovery.py init
```

5. Отредактировать созданный `recovery-plan.json`: использовать отдельный archive ID для каждого независимого контура и добавить только подтверждённые владельцем пути.
6. Просмотреть план перед первым backup. Сам факт просьбы настроить recovery разрешает создать план, но не разрешает угадывать или массово собирать секреты.

## Создать и проверить backup

```bash
python3 skills/encrypted-recovery/scripts/encrypted_recovery.py status
python3 skills/encrypted-recovery/scripts/encrypted_recovery.py backup
python3 skills/encrypted-recovery/scripts/encrypted_recovery.py verify
```

`age` сам запрашивает новый пароль и его подтверждение во время шифрования. Немедленная проверка просит пароль ещё раз. Передать терминал владельцу и не просить пароль в чате. В агентской сессии не оставлять ввод пустым: `age` может напечатать автоматически созданную парольную фразу в терминал. Не пытаться передавать пароль через аргументы, stdin, environment, shell history или файлы проекта: официальный парольный режим `age` намеренно работает через терминал.

После первого backup попросить владельца повторно выполнить `verify` с паролем, взятым из менеджера паролей. Это проверяет не только архив, но и сохранность независимой копии пароля. Не запускать password-backup в unattended cron: для автоматического режима нужен отдельный recipient/key workflow и отдельное решение владельца.

Retention не удаляет старые копии во время backup: локальная запись ещё не доказывает завершённую синхронизацию. После подтверждения внешней сохранности сначала выполнить dry run, затем явное удаление:

```bash
python3 skills/encrypted-recovery/scripts/encrypted_recovery.py prune
python3 skills/encrypted-recovery/scripts/encrypted_recovery.py prune --confirm-synced
```

## Восстановить

Сначала восстановить в отдельную ignored-папку:

```bash
python3 skills/encrypted-recovery/scripts/encrypted_recovery.py restore --archive workspace-secrets
```

Скрипт проверяет manifest и SHA-256 до записи результата. Просмотреть список файлов и только затем вручную перенести нужные значения в рабочие пути. Не добавлять автоматическое восстановление поверх существующих секретов и не выводить их содержимое в чат.

## Обновить allowlist

- При появлении нового постоянного ignored-секрета добавить его точный путь в подходящий scope `recovery-plan.json`.
- Не включать кэши, `node_modules`, браузерные профили, временные выгрузки и сырые логи без отдельного решения владельца.
- После изменения плана выполнить `status`, новый `backup` только изменённого scope через `--archive ID` и `verify --archive ID`.
- Коммитить план только после проверки, что сами значения секретов в нём отсутствуют.
