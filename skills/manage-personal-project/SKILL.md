# Ведение личного проекта

## Определить границы

1. Проверить существующие проекты и задачи.
2. Если в корне проекта есть `workspace-reference.json`, прочитать `localPath`, синхронизировать соответствующий отдельный Git checkout и продолжить работу в каноническом общем объекте. Не записывать содержательные изменения в reference-папку.
3. Создавать проект только при наличии долгосрочного контура, а не для одного небольшого вопроса.
4. Сформулировать назначение, входящие и исключённые темы, основной источник истины.

## Создать структуру

- Создать `projects/slug/` из `templates/project/`.
- В `README.md` держать устойчивую структуру и ссылки на реестры.
- В `status.md` держать датированный текущий срез, ближайший шаг и блокеры.
- Конкретные действия вести отдельными задачами и ссылать их на проект.

## Обновлять

- После завершения связанной задачи обновлять состояние проекта, если вывод меняет общую картину.
- Не дублировать одинаковое правило или факт в нескольких местах; назначить один источник истины и ссылаться на него.
- Отделять исторические записи от текущего статуса.

## Проверить

- Убедиться, что ссылки на активные задачи актуальны.
- Проверить дату статуса, следующий шаг и отсутствие противоречащих друг другу реестров.
- Просмотреть diff перед фиксацией результата.
