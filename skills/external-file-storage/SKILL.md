# Внешнее файловое хранилище

## Граница интеграции

- Работать только с локальной папкой, выбранной владельцем. Не предполагать конкретного облачного провайдера и не требовать его API.
- Не считать успешную локальную запись доказательством завершённой облачной синхронизации.
- Не менять доступ к облачной папке, режим общего доступа или аккаунт провайдера без отдельного поручения.
- Не удалять и не перезаписывать пользовательские файлы «заодно». Для замены значимого файла сначала подтвердить точный целевой путь.

## Настройка компьютера

1. Выполнить `python3 scripts/integration_preflight.py external-file-storage`.
2. Если папка не настроена, попросить владельца выбрать существующую локальную папку либо принять уже названный им путь.
3. Выполнить `python3 scripts/integration_preflight.py external-file-storage --setup --path ПАПКА`.
4. Убедиться, что проверка записи, чтения и удаления прошла. Абсолютный путь хранится только в ignored `.local/external-file-storage.json`.
5. На другом компьютере повторить настройку локального пути; tracked-конфигурацию с абсолютным путём не создавать.

Выбранная папка должна быть отделена от рабочей копии: нельзя выбирать сам Workspace, каталог внутри него или его родительский каталог. Это не позволяет внешним файлам случайно попасть в Git и не превращает домашнюю директорию в неявный корень хранилища.

## Размещение файлов

- Зеркалировать задачи как `tasks/<year>/<NNNN--slug>/`, а проекты как `projects/<slug>/` внутри выбранного корня.
- Для исходников использовать `source/`, для результатов – `outputs/`, для зашифрованных recovery-архивов – `recovery/` или `encrypted/`.
- Для резервирования ignored-секретов не копировать plaintext вручную: использовать `encrypted-recovery`, который пишет только аутентифицированно зашифрованный и проверенный архив.
- Для каждого необходимого файла вне Git фиксировать рядом с задачей или проектом переносимый `external-assets.yml`: относительный `storage_path`, ожидаемый ignored `local_path`, назначение, чувствительность, дату обновления и SHA-256, если он известен.
- Не записывать в tracked manifest абсолютный путь текущего компьютера, токены, cookies или персональные URL, которые не нужны для восстановления.

## Проверка и восстановление

- Перед работой с внешним файлом повторить обычный preflight. Он не создаёт файлов и не изменяет конфигурацию.
- Учитывать placeholder-файлы облачных клиентов: существующий путь ещё не гарантирует, что содержимое скачано локально.
- После копирования значимого файла проверить размер и SHA-256, если checksum указан в manifest.
- При конфликтной копии облачного клиента не выбирать победителя автоматически: сохранить обе версии и сообщить владельцу различия.
